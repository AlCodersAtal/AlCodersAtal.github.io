Wireless Communication Basics

Connectivity is the defining characteristic of IoT. Without communication, a smart sensor is just a sensor. Understanding your options helps you pick the right one for each project.

The Connectivity Landscape

IoT devices use many different wireless technologies. The choice depends on range, power, data rate, and cost:

TechnologyRangePowerData RateBest For
Wi-Fi50–100mHighUp to 600 MbpsHome IoT, data-heavy
Bluetooth/BLE10–100mLow1–3 MbpsWearables, beacons
Zigbee10–100mVery low250 kbpsSmart home mesh
Z-Wave30–100mLow100 kbpsSmart home security
LoRa / LoRaWAN2–15 kmVery low0.3–50 kbpsAgriculture, city IoT
4G/LTENationwideHighUp to 150 MbpsMobile, remote locations
5GNationwideMediumUp to 10 GbpsIndustrial, V2X, massive IoT
NB-IoTNationwideUltra low26 kbpsSmart meters, trackers

Choosing a Protocol — Quick Guide

  • Home/office IoT: Wi-Fi (if plugged in) or Zigbee/BLE (if battery-powered)
  • Wearables: Bluetooth LE
  • Agricultural / long-range sensing: LoRa or NB-IoT
  • Mobile asset tracking (vehicles): 4G/LTE or GPS + 4G
  • Learning IoT: Wi-Fi — most supported, easiest to debug

Wi-Fi for IoT

Wi-Fi is the most common IoT connectivity for home and office environments. The ESP32 has Wi-Fi built-in and the Arduino WiFi library makes it easy to use.

How Wi-Fi Works in IoT

Your ESP32 connects to a Wi-Fi router like a phone or laptop. Once connected, it gets an IP address and can communicate with any server on the internet — sending sensor data, fetching commands, or hosting a local web server.

Connecting ESP32 to Wi-Fi

Arduino C (ESP32)
#include <WiFi.h> const char* ssid = "YourNetwork"; const char* password = "YourPassword"; void setup() { Serial.begin(115200); WiFi.begin(ssid, password); Serial.print("Connecting"); while (WiFi.status() != WL_CONNECTED) { delay(500); Serial.print("."); } Serial.println("\n✅ Connected!"); Serial.println(WiFi.localIP()); }

ESP32 as a Web Server

Your ESP32 can host a tiny web server. Anyone on your local network can open a browser, type the ESP32's IP, and see a webpage — or send commands to the device.

Arduino C (ESP32)
#include <WiFi.h> #include <WebServer.h> WebServer server(80); void handleRoot() { server.send(200, "text/html", "<h1>Hello from ESP32!</h1>"); } void setup() { // ... WiFi connect code here ... server.on("/", handleRoot); server.begin(); Serial.println("Web server started!"); } void loop() { server.handleClient(); }

Wi-Fi Limitations in IoT

  • Power hungry: Wi-Fi draws 160–260 mA during transmission — batteries drain fast
  • Range: Typically 30–50m indoors
  • Router required: Needs local Wi-Fi infrastructure
💡 Deep Sleep Solution
Use ESP32's deep sleep mode for battery projects. The device sleeps consuming only 10 µA, wakes up every N minutes, sends data over Wi-Fi, then goes back to sleep.

Bluetooth & BLE

Bluetooth Low Energy (BLE) is the go-to for battery-powered devices that communicate with smartphones. The ESP32 has BLE built-in.

Classic Bluetooth vs BLE

FeatureClassic BluetoothBLE (Bluetooth Low Energy)
PowerHigh (1–100 mA)Very low (0.01–0.5 mA)
Data rate2–3 Mbps125 kbps – 2 Mbps
Range10–100m10–100m
Use caseAudio streaming, file transferSensors, wearables, beacons
Connection time~6 seconds~3 milliseconds
📱 IoT Choice
For IoT, BLE is almost always the right choice. It's what fitness bands, smart watches, glucose monitors, and smart home sensors use.

BLE Concepts: GATT, Services, Characteristics

BLE uses a client-server model called GATT (Generic Attribute Profile):

  • Server (ESP32): The device with data. Exposes "Services"
  • Client (phone/PC): Reads data from the server
  • Service: A group of related characteristics (like "Environmental Sensing Service")
  • Characteristic: A single data point with a value (like "Temperature")
  • UUID: Unique ID for each service and characteristic

ESP32 BLE Server Example

Arduino C (ESP32)
#include <BLEDevice.h> #include <BLEServer.h> #include <BLEUtils.h> #define SERVICE_UUID "12345678-1234-5678-1234-56789abcdef0" #define CHAR_UUID "abcdef01-1234-5678-1234-56789abcdef0" BLECharacteristic* pCharacteristic; void setup() { Serial.begin(115200); BLEDevice::init("ESP32-IoT-Sensor"); BLEServer* pServer = BLEDevice::createServer(); BLEService* pService = pServer->createService(SERVICE_UUID); pCharacteristic = pService->createCharacteristic( CHAR_UUID, BLECharacteristic::PROPERTY_READ ); pCharacteristic->setValue("Hello from ESP32!"); pService->start(); pServer->getAdvertising()->start(); Serial.println("BLE advertising... scan with nRF Connect app"); } void loop() { delay(1000); }

MQTT Protocol

MQTT (Message Queuing Telemetry Transport) is the most popular IoT protocol. It's lightweight, efficient, and designed specifically for constrained devices and unreliable networks.

What is MQTT?

MQTT is a publish-subscribe messaging protocol. Instead of devices talking directly to each other, they all communicate through a central broker:

  • Publisher: A device that sends (publishes) data to a topic
  • Subscriber: A device or app that listens (subscribes) to a topic
  • Broker: The server in the middle that routes messages (e.g., Mosquitto, HiveMQ)
  • Topic: A string address for messages (e.g., home/bedroom/temperature)

Why MQTT for IoT?

  • Tiny packet size — headers as small as 2 bytes
  • Works on slow, high-latency networks (3G, satellite)
  • QoS levels: 0 (fire and forget), 1 (at least once), 2 (exactly once)
  • Last Will message — broker notifies subscribers if a device disconnects unexpectedly
  • Retained messages — new subscribers get the last value immediately
📊 Scale
A single MQTT broker can handle millions of concurrent connections. Facebook Messenger was originally built on MQTT.

ESP32 MQTT Publisher

Arduino C (ESP32)
#include <WiFi.h> #include <PubSubClient.h> const char* ssid = "YourWiFi"; const char* password = "YourPassword"; const char* mqttServer = "broker.hivemq.com"; // Free public broker const int mqttPort = 1883; WiFiClient espClient; PubSubClient client(espClient); void setup() { Serial.begin(115200); WiFi.begin(ssid, password); while (WiFi.status() != WL_CONNECTED) delay(500); client.setServer(mqttServer, mqttPort); client.connect("ESP32Client"); Serial.println("Connected to MQTT broker!"); } void loop() { float temperature = 25.6; // Replace with real sensor read char msg[10]; dtostrf(temperature, 4, 1, msg); client.publish("alcoders/sensor/temp", msg); Serial.print("Published: "); Serial.println(msg); delay(5000); // Publish every 5 seconds }

Library needed: Install "PubSubClient" by Nick O'Leary in Arduino Library Manager.

HTTP & REST APIs in IoT

HTTP is the same protocol your browser uses. IoT devices can use HTTP to send data to cloud APIs — simple, universal, and no special broker needed.

HTTP in IoT

Your ESP32 can act as an HTTP client — making GET or POST requests to a server. This is how you send data to ThingSpeak, IFTTT, or any REST API.

  • GET request: Retrieve data from a server (or send simple values in URL)
  • POST request: Send data to a server (usually in JSON body)

Sending Data to ThingSpeak via HTTP

Arduino C (ESP32)
#include <WiFi.h> #include <HTTPClient.h> const char* ssid = "YourWiFi"; const char* pass = "YourPassword"; const char* apiKey = "YOUR_THINGSPEAK_API_KEY"; void sendToCloud(float temp, float humidity) { HTTPClient http; String url = "http://api.thingspeak.com/update?api_key="; url += apiKey; url += "&field1=" + String(temp); url += "&field2=" + String(humidity); http.begin(url); int code = http.GET(); Serial.print("HTTP response: "); Serial.println(code); http.end(); } void setup() { Serial.begin(115200); WiFi.begin(ssid, pass); while (WiFi.status() != WL_CONNECTED) delay(500); } void loop() { sendToCloud(25.5, 65.0); // Replace with real sensor values delay(15000); // ThingSpeak free tier: 15 sec minimum interval }

MQTT vs HTTP — When to Use Which

ScenarioBetter Choice
Frequent small data (every second)MQTT
Battery-powered, sleep cyclesHTTP (simpler, no persistent connection)
Two-way communication (device gets commands)MQTT
Sending to a simple REST APIHTTP
Many devices, central brokerMQTT
Learning / getting startedHTTP (easier to understand)
