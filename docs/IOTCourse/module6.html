Introduction to IoT Cloud Platforms

IoT devices generate enormous amounts of data. Cloud platforms give us somewhere to send, store, visualise, and act on that data — without running our own servers.

Why Cloud for IoT?

A microcontroller like the ESP32 has kilobytes of RAM. It can't store weeks of sensor readings or serve a web dashboard to hundreds of users. Cloud platforms solve this by providing:

  • Storage — time-series databases built for sensor data
  • APIs — simple HTTP or MQTT endpoints your device can push data to
  • Dashboards — ready-made charts and gauges, no frontend coding
  • Automation — trigger emails, SMS, or device commands when thresholds are crossed

Types of IoT Platforms

Device Management
Register, update, and monitor fleets of devices (AWS IoT Core, Azure IoT Hub)
Data Platforms
Ingest and visualise time-series data (ThingSpeak, InfluxDB + Grafana)
Maker Platforms
Quick prototyping with dashboards and automations (Adafruit IO, Blynk)
Edge Platforms
Run logic close to the device, sync to cloud (AWS Greengrass, Azure IoT Edge)

Choosing a Platform

ThingSpeak — Visualising Sensor Data

ThingSpeak is a free IoT analytics platform by MathWorks. Your ESP32 sends data via HTTP, and ThingSpeak plots it automatically. No server setup required.

Key Concepts

  • Channel — a collection of up to 8 data fields (e.g., temperature, humidity)
  • Write API Key — secret key your device uses to push data
  • Read API Key — key for fetching data (or make the channel public)
  • Update interval — free accounts: minimum 15 seconds between updates

Sending Data from ESP32

A simple HTTP GET request is all you need. Replace YOUR_API_KEY with your channel's Write API Key.

Arduino (ESP32)
#include <WiFi.h> #include <HTTPClient.h> #include <DHT.h> const char* ssid = "YourWiFi"; const char* password = "YourPass"; const char* apiKey = "YOUR_API_KEY"; #define DHTPIN 4 #define DHTTYPE DHT11 DHT dht(DHTPIN, DHTTYPE); void setup() { Serial.begin(115200); WiFi.begin(ssid, password); while (WiFi.status() != WL_CONNECTED) delay(500); dht.begin(); } void loop() { float t = dht.readTemperature(); float h = dht.readHumidity(); if (!isnan(t) && !isnan(h)) { HTTPClient http; String url = String("http://api.thingspeak.com/update?api_key=") + apiKey + "&field1=" + t + "&field2=" + h; http.begin(url); int code = http.GET(); Serial.println("HTTP: " + String(code)); http.end(); } delay(20000); // 20 seconds between updates }

Adafruit IO — Dashboards & Feeds

Adafruit IO is a maker-friendly cloud platform with drag-and-drop dashboards, feed history, and MQTT support. It's perfect for building interactive IoT projects.

Adafruit IO Concepts

Feed
A named data stream — like "temperature" or "button-state". Each feed stores a history of values.
Dashboard
A customisable web page of widgets — gauges, toggles, charts — connected to your feeds.
AIO Key
Your secret API key. Used in code to authenticate publishes and subscribes.
Triggers
Automated actions — send an email when a feed value exceeds a threshold.

Publishing to Adafruit IO via MQTT

Arduino (ESP32)
#include <WiFi.h> #include <Adafruit_MQTT.h> #include <Adafruit_MQTT_Client.h> #define AIO_SERVER "io.adafruit.com" #define AIO_PORT 1883 #define AIO_USERNAME "your_username" #define AIO_KEY "your_aio_key" WiFiClient client; Adafruit_MQTT_Client mqtt(&client, AIO_SERVER, AIO_PORT, AIO_USERNAME, AIO_KEY); Adafruit_MQTT_Publish tempFeed(&mqtt, AIO_USERNAME "/feeds/temperature"); void setup() { WiFi.begin("SSID", "PASS"); while (WiFi.status() != WL_CONNECTED) delay(500); } void loop() { if (!mqtt.connected()) mqtt.connect(); float temp = 28.5; // replace with real sensor read tempFeed.publish(temp); delay(10000); }

Building a Dashboard

  1. Log in to io.adafruit.com and create a Feed named temperature
  2. Go to Dashboards → New Dashboard
  3. Add a Gauge block, link it to your temperature feed
  4. Add a Line Chart to see history
  5. Run your ESP32 — watch the gauge update live!
Toggle Switches
Add a Toggle widget to your dashboard connected to a feed named "led". Subscribe to that feed on your ESP32 and you have wireless LED control with a beautiful UI.

Blynk — Mobile App for IoT

Blynk lets you build a custom smartphone app for your IoT project in minutes — no app development knowledge needed. Drag widgets onto a canvas and connect them to your hardware.

How Blynk Works

Blynk uses a cloud server as a relay between your phone app and your device. All communication goes through Virtual Pins — numbered data channels (V0, V1, V2...) that carry any value type.

  • Your ESP32 writes to Virtual Pin → phone app updates
  • User taps a button in app → ESP32 receives the command
  • No port forwarding, no static IP required

Blynk 2.0 Setup

  1. Create account at blynk.cloud
  2. Add a new Template → choose ESP32
  3. Create a Datastream (Virtual Pin V0, type: Double)
  4. Add a gauge widget to the web/mobile dashboard, link to V0
  5. Copy your Auth Token and Template ID

ESP32 Code with Blynk

Arduino (ESP32)
#define BLYNK_TEMPLATE_ID "TMPLxxxxxx" #define BLYNK_TEMPLATE_NAME "My IoT Project" #define BLYNK_AUTH_TOKEN "YourAuthToken" #include <BlynkSimpleEsp32.h> char ssid[] = "YourWiFi"; char pass[] = "YourPass"; BlynkTimer timer; void sendSensor() { float temp = 27.3; // replace with actual sensor Blynk.virtualWrite(V0, temp); } // Called when app writes to V1 (e.g. toggle button) BLYNK_WRITE(V1) { int val = param.asInt(); digitalWrite(2, val); // control onboard LED } void setup() { Serial.begin(115200); Blynk.begin(BLYNK_AUTH_TOKEN, ssid, pass); timer.setInterval(5000L, sendSensor); pinMode(2, OUTPUT); } void loop() { Blynk.run(); timer.run(); }
Blynk Free Tier Limits
The free Blynk plan allows 2 devices and limited datastreams. For classroom use, one account with shared credentials works, but avoid storing sensitive data.

AWS IoT Core — Enterprise IoT at Scale

AWS IoT Core is Amazon's managed IoT cloud service. It handles millions of devices, provides a powerful rules engine, and integrates with the full AWS ecosystem — databases, Lambda, machine learning, and more.

AWS IoT Core Architecture

Device Gateway
MQTT/HTTPS broker that accepts connections from devices worldwide
Device Registry
Database of all registered Things — metadata, status, certificates
Device Shadow
JSON document representing the last known state of a device, even when offline
Rules Engine
SQL-like rules that route incoming messages to DynamoDB, Lambda, S3, SNS, etc.

Security in AWS IoT

Unlike Adafruit IO's API key approach, AWS IoT uses mutual TLS (mTLS) — both the server and device present certificates. Each device gets:

  • A unique X.509 certificate (.crt file)
  • A private key (.key file)
  • The Amazon root CA certificate

These three files are embedded in your device firmware. No username/password needed — certificate presence is the authentication.

Device Shadow Example

A Device Shadow JSON looks like this. The "desired" state is what the cloud wants; "reported" is what the device last sent:

JSON
{ "state": { "desired": { "led": "on", "fanSpeed": 75 }, "reported": { "led": "off", "fanSpeed": 0, "temperature": 32.1 } } }

When the device comes back online, it reads the shadow, sees the delta (desired ≠ reported), and acts accordingly — then updates "reported" to match.

When to Use AWS IoT
AWS IoT is overkill for a classroom project, but essential for a real product. If you're shipping 1,000+ devices to customers, you need device management, OTA firmware updates, audit logging, and scalable ingestion — all of which AWS IoT provides.
What problem does the AWS IoT "Device Shadow" solve?
It backs up device firmware to S3
It mirrors device data to a second region
It stores the last known device state so commands work even when the device is offline
It provides a graphical UI for the device
