Why IoT Security Matters

IoT devices sit at the intersection of the physical and digital worlds. A compromised thermostat can unlock doors. A hacked industrial sensor can shut down a factory. Security is not optional.

The IoT Attack Surface

Every IoT device is potentially an entry point. The attack surface includes:

Device Firmware
Unencrypted or unverified firmware can be replaced with malicious code
Network Traffic
Unencrypted MQTT or HTTP traffic can be intercepted (man-in-the-middle)
Cloud APIs
Weak API keys or no authentication allows anyone to inject fake data
Physical Access
Devices in public places can be extracted, probed, or cloned

Real-World IoT Attacks

Casino Fish Tank (2017)Hackers entered a casino network through an internet-connected aquarium thermometerIsolate IoT on separate networks

Security Principles for IoT

  • Least privilege — devices should only have access to what they need
  • Defence in depth — multiple layers: encryption + auth + network segmentation
  • Fail secure — if something goes wrong, default to the safe state
  • Update regularly — firmware updates patch newly discovered vulnerabilities
Default Passwords Kill
The number one IoT security mistake is leaving default usernames and passwords unchanged. "admin/admin" and "root/root" are the first credentials every attacker tries.

Man-in-the-Middle Attack

When your device sends data over plain HTTP or unencrypted MQTT, anyone on the same network can intercept and read it — or inject fake data.

The fix is TLS (Transport Layer Security), the same encryption used by HTTPS websites. With TLS, even if traffic is intercepted, it can't be read or modified.

Hardcoded Credentials in Code

Never hardcode passwords directly in source code that might be shared:

Arduino — BAD Practice
// ❌ BAD: password visible in source code const char* password = "mySecretPass123"; // ✅ BETTER: store in a separate file not committed to Git #include "secrets.h" // listed in .gitignore // secrets.h contains: const char* password = "mySecretPass123";
Don't Upload API Keys to GitHub
Add secrets.h to your .gitignore file. Thousands of API keys are accidentally exposed on GitHub every day, and bots scan for them within minutes.

Encryption Basics for IoT

Encryption scrambles data so only the intended recipient can read it. Two forms matter most for IoT: encryption in transit (TLS) and encryption at rest (AES on storage).

Symmetric vs Asymmetric Encryption

Symmetric (AES)Asymmetric (RSA / ECC)
KeysOne shared secret keyPublic key + private key pair
SpeedVery fastSlower
Use in IoTEncrypting stored dataTLS handshake, certificates
ProblemHow to share the key securely?Computationally heavier

TLS actually uses both: asymmetric encryption to securely exchange a shared key, then symmetric (AES) for the actual data — best of both worlds.

MQTT over TLS (MQTTS)

To use encrypted MQTT on ESP32, connect to port 8883 instead of 1883, and provide the server's CA certificate:

Arduino (ESP32)
#include <WiFiClientSecure.h> #include <PubSubClient.h> const char* caCert = R"EOF( -----BEGIN CERTIFICATE----- ... (paste your CA cert here) ... -----END CERTIFICATE----- )EOF"; WiFiClientSecure secureClient; PubSubClient mqtt(secureClient); void setup() { secureClient.setCACert(caCert); mqtt.setServer("your-broker.com", 8883); // TLS port mqtt.connect("esp32-device"); }

Encrypting Stored Data with AES

If you log sensitive data to an SD card, consider encrypting it. The AESLib Arduino library provides AES-128/256 encryption:

Arduino
#include <AESLib.h> AESLib aesLib; byte aesKey[] = {0x00, 0x01, 0x02, ..., 0x0F}; // 16-byte key byte aesIV[] = {0x00, 0x00, 0x00, ..., 0x00}; // 16-byte IV char plaintext[] = "Temp=28.5,Hum=65"; char ciphertext[32]; aesLib.encrypt((byte*)plaintext, strlen(plaintext), (byte*)ciphertext, aesKey, 128, aesIV);
HTTPS is Already Encrypted
When you use HTTPClient to call https:// URLs in Arduino, you're already using TLS. Always prefer https:// over http:// for any API calls — especially those sending sensitive data.

Secure Firmware & OTA Updates

Once a device is deployed, how do you fix security vulnerabilities? Over-the-Air (OTA) updates let you push new firmware without physical access — but the update mechanism itself must be secured.

Why OTA Updates Matter

  • Security patches can be pushed without recalling devices
  • New features can be added after deployment
  • Bug fixes reach every device simultaneously
  • Physical access to thousands of deployed devices is impractical

Basic OTA Update on ESP32

The Arduino ArduinoOTA library lets your ESP32 accept firmware uploads over Wi-Fi from the Arduino IDE:

Arduino (ESP32)
#include <ArduinoOTA.h> #include <WiFi.h> void setup() { WiFi.begin("SSID", "PASS"); while (WiFi.status() != WL_CONNECTED) delay(500); ArduinoOTA.setPassword("ota_secret_pw"); // protect OTA access ArduinoOTA.setHostname("my-iot-device"); ArduinoOTA.begin(); } void loop() { ArduinoOTA.handle(); // must call in loop to check for updates // ... rest of your code }

Secure OTA Best Practices

PracticeWhy

Unsigned Firmware is Dangerous
If your OTA update doesn't verify a cryptographic signature, an attacker who controls the network can serve malicious firmware to your device. For production devices, always sign firmware with a private key and verify on-device before applying.

IoT Security Best Practices

Security isn't a single feature — it's a mindset applied throughout design, development, and deployment. This lesson consolidates the essential habits every IoT builder should follow.

The IoT Security Checklist

CategoryAction
CredentialsChange all default passwords. Use strong, unique passwords.
CredentialsNever hardcode passwords in source code. Use secrets.h + .gitignore.
TransportUse HTTPS / MQTTS. Never plain HTTP or unencrypted MQTT for sensitive data.
UpdatesImplement OTA updates with a password and firmware signature.
NetworkPut IoT devices on a separate VLAN/guest network, isolated from computers.
PortsClose all unused ports. Only open what's needed.
DataCollect only the data you need. Delete when no longer needed.
PhysicalDisable debug ports (UART, JTAG) in production firmware.

Network Segmentation

One of the most effective IoT security measures costs nothing: put your IoT devices on a separate Wi-Fi network (most home routers have a "guest network" feature).

  • If an IoT device is compromised, attackers can't reach your laptop or NAS
  • Limits the blast radius of any single breach
  • Easier to monitor unusual traffic from a dedicated IoT VLAN

Privacy by Design

Data minimisation — collecting only what you need — is both a security practice and a legal requirement under regulations like GDPR. Before adding a sensor, ask:

  • Do we actually need this data?
  • How long should we keep it?
  • Who else can access it?
  • What happens if it leaks?
Threat Modelling
Before building any IoT system, spend 15 minutes asking: "Who might want to attack this? What could they gain? What would it cost them?" This simple exercise surfaces the biggest risks before any code is written.
